# Plan 2b-3 design — installer orchestration (hardened via adversarial workflow)

Date: 2026-06-25  ·  Source: 5-agent design+verify+synthesize workflow (wf_27a59c3d-7cf)

## Overview
Plan 2b-3 adds the installer/orchestration half of the clone-and-run wizard, split into a pure plan/detect/template core (fully unit-testable in-sandbox with mocked subprocess/httpx) and a thin execution/stream shell (real-Mac only). A new `app/install/` package holds: `steps.py` (ordered Step registry), `detect.py` (pure parsers + thin probes resolving the brew prefix and a *required* Python 3.12 interpreter), `commands.py`/`plan.py` (pure argv builders), `riffado.py` (pure secret-gen + .env render reusing `envfile.upsert`), `launchd.py` (100% pure plist templates for the worker schedule and a NEW always-on KeepAlive web agent), and `exec.py`/`runner.py` (the only modules that Popen/launchctl). A new `app/install_api.py` router mounts under `/api/install` and adds SSE streaming via a threadpool+queue bridge so long pip/docker pulls never block uvicorn's event loop and survive tab-close. The ML stack installs into a DEDICATED `worker/.venv-ml` (never the live server `worker/.venv`), the worker plist + sync_and_reconcile point at it, and every step is detect-gated for idempotent re-runs. The wizard UI gains a multi-step stepper (`install.html`/`install.js`) with live log consoles, exact-command Copy+Retry on failure, and guide+Test panels for the two human/GUI steps (Docker Desktop, Plaud OTP).

## Locked decisions
- Adopt Design 1 as the baseline; reject D2/D3's reuse-and-recreate of worker/.venv — the ML stack goes into a dedicated worker/.venv-ml so the wizard never deletes the venv the live uvicorn server is executing from.
- Strictly separate a PURE layer (steps/detect-parsers/commands/plan/riffado-secrets/launchd-templates/sse-format/run_step-control-flow against a FakeRunner) from a THIN real-Mac shell (Popen, launchctl, brew/pip/docker binaries). Pure layer is the test target.
- Pin Python 3.12 by REQUIRING it (parse `Python 3.12.x`), never falling back to the user's python3 (=3.14, no torch/pyannote wheels); if absent emit `brew install python@3.12` as a prerequisite command in the same step.
- Detect completeness by importing mlx_whisper, torch, pyannote.audio in .venv-ml — not by venv existence — so a half-finished pip is treated as incomplete and re-run.
- Generate Riffado secrets with stdlib `secrets.token_hex` (32/32/24 bytes) — no openssl subprocess; reuse existing .env values only when NON-EMPTY (key-present is insufficient) so a running pgdata volume's POSTGRES_PASSWORD is never rotated.
- Write deploy/riffado/.env via the existing envfile.upsert (already chmod 0600), preserving RIFFADO_VERSION/APP_URL/DISABLE_REGISTRATION; secrets go through a separate POST endpoint so they never transit the SSE log.
- Declare Riffado up only after `docker compose up -d --wait` succeeds OR a conn_check.probe poll of http://127.0.0.1:3000/api/health returns 200 — never on bare `up -d`. Gate the Riffado step behind a live `docker info` check; if the daemon is down, emit a guide-style 'Start Docker Desktop, then Retry' instead of streaming a cryptic daemon error.
- Resolve the real brew prefix (/opt/homebrew confirmed on this Mac, arm64-gated) at plist-generation time and inject {prefix}/bin:{prefix}/sbin into the plist PATH as a parameter — golden-tested — fixing the documented launchd ffmpeg-not-found silent failure.
- NEW always-on web agent uses KeepAlive as a DICT {SuccessfulExit:false, Crashed:true} + ThrottleInterval:10 (not bare KeepAlive=true) to avoid a crash-loop when port 8787 is held; mirror ./run's argv/WorkingDirectory={repo}/app/PYTHONPATH={repo}:{repo}/worker/src exactly so launchd and manual launches behave identically.
- Before bootstrapping the web agent, probe port 8787 with socket.connect_ex; if held by the foreground ./run, WARN ('close the terminal tab; the background service takes over') rather than killing the live server the wizard is being served from.
- launchd reload = `bootout` (swallow not-loaded) then `bootstrap gui/$UID` (modern macOS form); install_argv is pure and unit-tested as a two-command sequence.
- SSE runs the blocking subprocess in a threadpool bridged by a queue.Queue with a 15s heartbeat comment; the subprocess OUTLIVES the request (tab close does not abort a 4-minute torch install); reconnect re-detects and shows skip or re-streams.
- Error/terminal SSE frames carry only shlex.join(argv) + exit code — never env — so a failed pip/docker line cannot leak a key; error frames drive the UI Copy + Retry box.
- Treat absent Homebrew as guide+Test (like Docker), not silent auto-install: the brew installer needs sudo/Xcode-CLT and prompts for a TTY password the SSE stream cannot supply, which would hang invisibly. On THIS machine brew is already present so the step skips.
- Mount install_router in app/server.py alongside setup_router; reuse the existing monkeypatch/TestClient + _FakeHttpx test patterns; tests land in app/tests/ (pythonpath already ['..','../worker/src']).

## Tasks (ordered)

### Task 1: Scaffold app/install/ package + pure Step registry and dataclasses  (testable here)
Files: app/install/__init__.py, app/install/steps.py, app/tests/test_install_steps.py
Define frozen `Step` dataclass (id, title, kind in {auto,guide}, detect, plan, guide_url) and the ordered `ALL_STEPS` list: brew -> ffmpeg -> py312 -> ml -> docker(guide) -> riffado -> plaud_otp(guide) -> launchd. Pure module importing nothing that shells out. Tests assert ordering, ids, kind classification (docker/plaud_otp/brew-absent are guide), and that every auto step exposes a plan callable.
Risks addressed: wrong step ordering / Riffado not gated behind Docker; brew sudo/TTY hang -> brew classified guide when absent

### Task 2: Pure detection parsers + thin probes (brew prefix, REQUIRED py3.12, ml-import, docker info, compose health)  (testable here)
Files: app/install/detect.py, app/tests/test_install_detect.py
Pure parsers fed captured strings: is_py312('Python 3.12.13')->True / rejects 3.14, parse_brew_prefix, parse_docker_info(rc/json), parse_compose_ps(json), ml_imports_ok(import-check rc). Thin probes (real-Mac, injected runner): find_brew() (/opt/homebrew/bin/brew first, then which), find_python312() probing brew --prefix python@3.12 then /opt/homebrew/opt/python@3.12/bin/python3.12 then which python3.12 — REQUIRES 3.12, returns None (never python3) if absent. venv_python_for_ml()->worker/.venv-ml/bin/python. Tests cover every parser with canned output incl. the 3.14-reject and three docker states (daemon-up/app-missing/app-not-running).
Risks addressed: python3=3.14 has no torch/pyannote wheels -> require 3.12, never fall back; half-finished pip looks installed -> detect by import not venv existence; docker daemon down vs app missing -> distinct guide copy; brew prefix drift -> resolve real prefix

### Task 3: Pure argv builders + plan constructors (brew/ffmpeg/py312, .venv-ml create+pip, docker compose --wait)  (testable here)
Files: app/install/commands.py, app/install/plan.py, app/tests/test_install_commands.py
Pure functions returning list[str] argv (never shell strings): brew_install_ffmpeg, brew_install_py312, make_ml_venv(py312)->[py312,'-m','venv','worker/.venv-ml'], pip_upgrade(venvpy), pip_install_ml(venvpy)->['-m','pip','install','-r','worker/requirements-ml.txt'], docker_info(), compose_up(dir)->['docker','compose','-f',<compose>,'up','-d','--wait']. plan(step,env) composes per-step command lists, prepending brew_install_py312 when find_python312 is None. Env carries repo_root + resolved brew prefix so tests inject fakes. Asserted as exact argv lists.
Risks addressed: ML stack must target .venv-ml not the live .venv; compose up -d races migrations -> use --wait; py312 prerequisite auto-prepended when absent; no-shell argv (injection-safe)

### Task 4: Riffado secret-gen + .env render (pure) reusing envfile.upsert; non-empty reuse contract  (testable here)
Files: app/install/riffado.py, app/tests/test_riffado_secrets.py
gen_secrets(rng=secrets.token_hex) -> {BETTER_AUTH_SECRET: hex(32), ENCRYPTION_KEY: hex(32), POSTGRES_PASSWORD: hex(24)} (injectable RNG for determinism). missing_secret_keys(existing_text) returns keys whose VALUE is blank (not just absent). write_env_idempotent() fills only blank keys via envfile.upsert against deploy/riffado/.env (inherits chmod 0600), preserving RIFFADO_VERSION/APP_URL/DISABLE_REGISTRATION. Tests assert key lengths/hex charset, that a fully-populated .env yields no missing keys (no rotation), and that an empty-value line IS treated as missing.
Risks addressed: rotating POSTGRES_PASSWORD orphans pgdata -> reuse only non-empty values; openssl subprocess failure mode removed (stdlib secrets); secrets file must be 0600; secrets never echoed (separate path)

### Task 5: Pure launchd plist templates: worker schedule (-> .venv-ml) + NEW always-on web agent  (testable here)
Files: app/install/launchd.py, deploy/launchd/com.example.plaudautomation.web.plist, app/tests/test_launchd_templates.py
render_worker_plist(repo, ml_python, brew_prefix) and render_web_plist(repo, server_python, brew_prefix, port) return plist XML strings via plistlib.dumps (valid XML, XML-escaped paths). Worker: ProgramArguments=[{repo}/worker/.venv-ml/bin/python, scripts/sync_and_reconcile.py], WorkingDirectory={repo}/worker, PYTHONPATH=src, RunAtLoad, StartInterval=1800, PATH={prefix}/bin:{prefix}/sbin:/usr/bin:/bin:/usr/sbin:/sbin, logs->state/automation.log, ProcessType=Background. Web: uvicorn server:app on 127.0.0.1:port, WorkingDirectory={repo}/app, PYTHONPATH={repo}:{repo}/worker/src, RunAtLoad, KeepAlive DICT {SuccessfulExit:false,Crashed:true}, ThrottleInterval=10, logs->state/web.log, ProcessType=Interactive. Pure install_argv(label,path) -> [['launchctl','bootout',...],['launchctl','bootstrap','gui/$UID',path]]. Golden-tested; the committed .web.plist is the documented template fallback.
Risks addressed: launchd PATH omits Homebrew -> ffmpeg silent failure (brew prefix param); KeepAlive=true crash-loops on port-held -> KeepAlive dict + ThrottleInterval; launchd/manual drift -> argv mirrors ./run exactly; worker plist must point at .venv-ml not .venv; re-load errors -> bootout-then-bootstrap idempotent

### Task 6: SSE formatter (pure) + run_step control flow against a FakeRunner  (testable here)
Files: app/install/runner.py, app/install/orchestrate.py, app/tests/test_install_orchestrate.py
runner.py defines Runner Protocol (stream/capture) + a FakeRunner (test) seeded with scripted (rc, lines) per argv; RealRunner (Popen, real-Mac) is a thin stub here, fully exercised on-Mac. sse_format(event,data)->'event: x\ndata: {json}\n\n' (pure, tested). orchestrate.run_step(step, runner, emit): (1) detect -> if satisfied emit skip+done (idempotent); (2) for each command stream lines via runner -> emit log; (3) on non-zero exit emit error{cmd:shlex.join(argv),code} carrying ONLY argv (no env) and stop. Tests drive the full skip/sequence/stop-on-error/error-carries-exact-command paths with FakeRunner — zero real subprocess.
Risks addressed: idempotent re-run -> detect-gate skip; exact failed command for Copy+Retry; no env/secret in error frames (shlex.join argv only); control flow testable without a Mac

### Task 7: install_api router: /status, SSE /stream (threadpool+queue), /riffado/secrets, /test/docker, /launchd/load; mount in server.py  (testable here)
Files: app/install_api.py, app/server.py, app/tests/test_install_api.py
New APIRouter prefix=/api/install: GET /status runs all detect probes -> per-step {done,detail,kind} for resume; GET /stream/{step_id} -> StreamingResponse(text/event-stream) running the blocking subprocess in a threadpool bridged by queue.Queue with a 15s heartbeat ':\n\n' (so it never blocks the event loop and the subprocess outlives client disconnect); POST /riffado/secrets generates+writes .env (separate from streaming); GET /launchd + POST /launchd/load render+install plists with a pre-bootstrap socket.connect_ex(127.0.0.1,8787) probe that returns a warn flag instead of fighting foreground ./run; POST /test/docker -> parse_docker_info. Mount install_router in server.py next to setup_router. Tests override the Runner via Depends with a FakeRunner: assert status JSON, SSE framing/skip/error-carries-argv, secrets-never-in-stream, docker-down gates Riffado, launchd warn-on-port-held. Real Popen/launchctl are the only untested surface.
Risks addressed: blocking Popen freezes uvicorn -> threadpool+queue bridge; tab close kills install -> subprocess outlives request, reconnect re-detects; long pip/pull idle-timeout -> heartbeat; wizard kills its own server -> port probe warns not kills; secrets in log -> separate secrets endpoint; daemon-down cryptic error -> docker info gate

### Task 8: Multi-step wizard UI: stepper, live log consoles, Copy+Retry, guide+Test panels  (testable here)
Files: app/web/install.html, app/web/install.js, app/web/wizard.html, app/web/style.css, app/tests/test_install_static.py
Render install steps as a vertical stepper from GET /api/install/status BEFORE the existing destination/provider/keys fieldsets (which stay unchanged). Each auto step: Install/Retry opens new EventSource('/api/install/stream/'+id), appends log frames to a <pre> console, flips badge on done/skip; on error shows a red box with the exact command + Copy (navigator.clipboard) + Retry. Guide steps (Docker, Plaud OTP) render guide_url + the existing Test buttons (reuse /api/setup/test/riffado, add /api/install/test/docker). A 'Generate & load background services' button POSTs /api/install/launchd/load and surfaces the port-held warning; a 'show me the command' affordance reveals the pure-generated plist + launchctl lines. On reload the stepper rehydrates from /status (resume at first incomplete step). Static test asserts the served HTML/JS reference the new endpoints and Copy/Retry hooks (DOM-level only; real EventSource/clipboard is browser-only).
Risks addressed: resume at first incomplete step (no client state); exact-command Copy+Retry surfaced in UI; guide steps never open an SSE stream (no deadlock); manual-command fallback for launchd

### Task 9: Point worker plist + sync_and_reconcile at .venv-ml; templated worker plist source  (testable here)
Files: deploy/launchd/com.example.plaudautomation.plist, worker/requirements-ml.txt, app/tests/test_launchd_worker_path.py
Update the committed worker plist comments/placeholders to document .venv-ml (the generated plist is authoritative; this file is the fallback template). Confirm sync_and_reconcile.py needs no code change (it runs under whatever interpreter launchd invokes) but is now driven by .venv-ml/bin/python in the generated plist — assert via a launchd template test that the worker plist's ProgramArguments[0] ends with worker/.venv-ml/bin/python and not worker/.venv/bin/python. No new runtime egress; pure doc/template + golden assertion.
Risks addressed: ML stack venv must never collide with the live server .venv; worker runs ML code so must use the 3.12 ML interpreter

## Real-Mac-only validation
- RealRunner.stream/capture actually Popen-ing brew/ffmpeg/pip/docker (the only impure exec surface).
- `brew install python@3.12` and the ML pip install of mlx-whisper/pyannote.audio/torch/silero-vad into worker/.venv-ml (heavy; needs real wheels + Apple MLX/MPS).
- Docker Desktop install (GUI .app — guide+Test only) and its first-launch license/privacy modal that keeps `docker info` red until clicked through.
- `docker compose up -d --wait` pulling the riffado image + postgres:16-alpine (hundreds of MB) and running first-boot DB migrations.
- Plaud email-OTP login into the Riffado web UI at http://127.0.0.1:3000 (human step) and the subsequent op_… API-key paste, validated by check_riffado hitting /api/v1/recordings.
- launchctl bootout/bootstrap of both agents and the ~/Library/LaunchAgents/*.plist writes; KeepAlive/ThrottleInterval crash-loop behavior; the worker schedule actually firing every 30 min + at load.
- The always-on web agent surviving terminal close / logout / crash and the real port-8787 collision with a foreground ./run (binds once ./run exits).
- Browser-only behavior: EventSource SSE rendering of a minutes-long install, navigator.clipboard Copy, and tab-close-then-reconnect re-detect.
- Homebrew absent path: the NONINTERACTIVE brew install needing sudo/Xcode CLT and prompting for a TTY password the SSE stream cannot supply (why brew-absent is guide+Test).

## Open risks
- ./run still builds worker/.venv from the user's python3 on a clean clone; if that default is 3.14 the SERVER venv is 3.14. FastAPI/uvicorn/httpx have 3.14 wheels (fine), but pydantic/httpx native builds can occasionally lag a new CPython — low risk, worth a smoke check on the recipient's Mac.
- find_python312 depends on `brew install python@3.12` having run; if Homebrew itself is absent and classified guide+Test, the ML step is blocked until the user installs brew manually — the stepper must clearly order brew before py312/ml.
- socket.connect_ex(127.0.0.1,8787) cannot distinguish 'our own already-loaded web agent' from a foreign listener with certainty; the warn-vs-load decision is heuristic and only fully validated on a real Mac with both ./run and the agent present.
- compose health gate polls /api/health, which is up before any user exists; the Plaud-OTP login + op_ key are a strictly later human gate (check_riffado /api/v1/recordings) — the wizard must not declare the Riffado step 'done' on health alone if it also needs the API key for the worker.
- Riffado 0.5.6 image tag and the ghcr.io/riffado/riffado registry must remain pullable; a yanked tag would fail compose up with an error the SSE surfaces as Copy+Retry but cannot self-heal.
- Two launchd agents both reference worker/state/*.log (automation.log vs web.log) — distinct files as designed, but state_dir resolution must stay consistent between ./run, the plists, and the worker (WORKER_STATE_DIR override) or logs/DBs could split across locations.
- Full scope (8-9 tasks) may be large for one plan; tasks 1-7 are a coherent testable-here first plan, tasks 8 (UI) and 9 (plist repoint) plus all real-Mac validation can be a second plan if needed.
